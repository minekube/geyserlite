# Architecture

## High level

```
Bedrock client ─UDP 19132─▶ libgeyserlite.so / geyserlite native executable
                                │
                                │ Floodgate AES-128 (loopback or TCP)
                                ▼
                        Java MC server / proxy
```

A single GraalVM `native-image` build of upstream
[GeyserMC/Geyser](https://github.com/GeyserMC/Geyser) compiled into two
artifacts:

- **`geyserlite` native executable** — drop-in for `Geyser-Standalone.jar`.
  Used by Docker consumers and by the Go/Rust libraries' subprocess fallback.
- **`libgeyserlite.so`** — same code built with `--shared`, exposing
  `@CEntryPoint`-annotated lifecycle functions. Loaded via `dlopen` by the Go
  (`purego`) and Rust (`libloading`) libraries for in-process embedding.

Both ship from a single CI run, so they always agree on Geyser version,
flags, and reflection metadata.

## Why native, why these flags

The full memory journey is in [`../ROADMAP.md`](../ROADMAP.md). Headlines:

- JVM Geyser idles at ~440 MB RSS — too big for a 256 MB Fly.io VM.
- `native-image` with default flags cuts that to ~220 MB.
- Tuning the Netty allocator, capping direct memory, lowering thread counts,
  and using `--strict-image-heap` + `--static --libc=musl` brings it to
  **~80 MB RSS** under load with the binary at **~107 MB**.
- All flags live in [`../build/flags.sh`](../build/flags.sh), each annotated.

## Soft fork without contributions

We don't fork or contribute upstream. Our build pipeline:

1. Clones `GeyserMC/Geyser` at the SHA pinned in `build/geyser.version`.
2. Layers in `build/overlay/` — pure additions, never overwrites.
3. Applies stable source changes by intent, including registering the overlay
   subproject and adapting the standalone bootstrap for embedding.
4. Applies `build/patches/*.patch` for minimal contextual source changes.
5. Runs `native-image` twice on Linux: once for the executable, once for the
   `.so`. Windows CI builds and smokes the subprocess `.exe` artifact.

Renovate watches upstream master and opens a PR bumping `geyser.version` on
new commits. CI re-applies overlay + patches; conflicts surface as failed
PRs for human attention. The `apply-overlay.sh` and `patches/README.md`
in `../build/` document the maintenance flow.

## Two modes from the language libraries

```
Go program / Rust program
        │
        ├── Mode::Embedded (default)
        │       │
        │       ▼
        │   purego.Dlopen / libloading::Library
        │       │
        │       ▼
        │   libgeyserlite.so   (in process)
        │
        └── Mode::Subprocess (opt-in)
                │
                ▼
            os/exec / tokio::process
                │
                ▼
            geyserlite native executable
                    (separate process)
```

**Embedded mode** is the default because it's simpler operationally: one
process, no subprocess management, no signal forwarding, no IPC. The price
is shared crash blast-radius — a Geyser native crash kills the host. We
accept that trade because crashes in the shipped GraalVM build of Geyser are
rare enough to be tolerable, and Fly's auto-restart catches them.

**Subprocess mode** exists for users who can't accept that trade. Same API,
different mode flag. Crash-isolated; restart-on-crash with exponential
backoff; signal forwarding; structured log capture.

## C ABI, language-agnostic

The shared library exports a small C ABI:

```c
int geyser_create_isolate(IsolateThread **out_thread);
int geyser_init(IsolateThread *thread, const char *config_path);
int geyser_run(IsolateThread *thread);              // blocks until shutdown
int geyser_shutdown(IsolateThread *thread);
int geyser_status(IsolateThread *thread);
int geyser_tear_down_isolate(IsolateThread *thread);
```

The full header (`libgeyserlite.h`) is auto-generated by GraalVM and
published as a release asset. Anyone can write bindings in any language
with a C FFI. Go and Rust are the two we maintain in-tree.

## What's not in scope

- Multiple Geyser instances per process. Substrate VM's isolate model would
  technically allow it, but no concrete use case yet justifies the extra
  state-management complexity.
- WASM. Java-to-WASI server-side isn't production-grade in 2026 (verified
  against current spec status). Maybe in 2-3 years.
- Live config reload, hot Floodgate key rotation, dynamic reconfiguration.
  Restart is fine; revisit when there's demand.
